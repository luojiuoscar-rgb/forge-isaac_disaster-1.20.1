# Isaac Disaster 项目记忆

更新时间：2026-06-30

这是 Codex 后续协作时应优先读取的项目记忆。它不是完整代码评审，而是基于一次结构扫描和抽样阅读整理出的系统地图、代码习惯、扩展点和改进线索。项目 Java 文件很多，`registries/ability/**/impl` 和 `registries/ability_effect/**/impl` 下大量 implementation 类模式相近，后续通常不需要全部读完，先读基类和 2-3 个同类样本即可。

## 项目定位

- Minecraft Forge 1.20.1 模组，mod id 是 `isaac_disaster`。
- 目标是把《以撒的结合》里的道具、饰品、胶囊、卡牌、道具池、底座、箱子、攻击方式等机制还原到 MC。
- 使用 Java 17、ForgeGradle、Parchment mappings。
- 依赖 JEI API/runtime、Curios runtime/API。
- `README.md`、Java 注释和部分文档是 UTF-8 中文，PowerShell 默认输出可能乱码；读中文内容时用 UTF-8 输出编码。

## 入口和注册骨架

主入口是 `src/main/java/net/luojiuoscar/isaac_disaster/IsaacDisaster.java`：

- 注册 `ModRegistries` 自定义注册表。
- 注册属性、物品、被动/主动/饰品/胶囊、创造栏、方块、方块实体、实体、声音、效果、loot type、loot modifier、gamerule、网络包、配置。
- 真正逻辑不在主类里，主类主要负责串注册。

自定义注册表核心在 `registries/ModRegistries.java`：

- `attack_trajectory`
- `bullet_color`
- `trigger_module`
- `recursive_module`
- `passive_ability`
- `active_ability`
- `trinket_ability`
- `set_ability`
- `pickup_ability`
- `attack_type`
- `attack_combination_rule`
- `executable_effect`

这组 registry 是附属模组兼容性的核心。附属模组理论上可以注册新的能力、效果、模块、攻击类型等，但当前很多系统仍依赖本项目自己的 `ItemId` / `TrinketId` enum 和静态列表，这一点会限制外部扩展的舒适度。

## 物品与能力系统

基类层：

- `IsaacItemAbility`：主动/被动/饰品能力的共同基类，保存 `id`、`level`，提供 `getDesc`、`getExtraDesc`、`getSynergyDesc`。
- `PassiveAbility`：提供 `onObtain`、`onRemove` 生命周期，内部先发自定义事件，再调用 `handleObtain` / `handleFirstObtain` / `handleRemove`。可用 `triggerEffect` 执行 `IExecutableEffect`。
- `ActiveAbility`：核心字段是 `CompositeTrigger trigger`，`onUse` 构造 `ExecutableEffectContext` 后 fire；`Car Battery` 通过 amplifier 改变效果强度。
- `PickupAbility`：卡牌、心、炸弹、电池等消耗品能力，使用 `CompositeTrigger`，默认使用后 shrink。
- `TrinketAbility`：提供 `onFirstEquipped`、`onEquipped`、`onUnequipped`。
- `SetAbility`：套装能力，使用 `requirementCount`，在被动道具计数跨过阈值时触发 obtain/remove。

物品层：

- `item/item/IsaacItem.java`：主动和被动物品共有 tooltip、稀有度颜色、`has_been_used`、`pool` NBT。
- `PassiveItem.java`：可 Curios 装备，也可在启用 `EXTRA_PASSIVE_ITEM_BACKPACK` 时右键进入额外背包；获得后触发能力。
- `ActiveItem.java`：用耐久模拟充能，`DAMAGE_PER_CHARGE_RATE = 100`；支持 `OverCharged` NBT；使用前发 `ActiveItemUseEvent`，成功后触发能力和记录。
- `Trinket.java`：Curios 饰品，带 `enchanted`、`swallowing`、`consumed` NBT；吞下后仍可长期生效。
- `Pill.java` / `GoldenPill.java`：胶囊物品入口，效果由 `PillEffectManager` 映射。

implementation 常见模式：

- 简单被动：`handleObtain` 调 `StatManager` 或添加 attack type/module，`handleRemove` 做反向操作。
- 主动：定义静态 `CompositeTrigger`，通常是一到多个 `SimpleTrigger(ModTriggerTypes.EMTPY, ModExecutableEffects.X)`。
- 饰品：很多 `onEquipped/onUnequipped` 是空实现，实际效果通过事件或 helper 查询吞下/装备列表。
- 复杂道具：可能在事件类中有静态 hook，例如 `GlitchedCrown.addDisplayItem`、`BingeEater.addDisplayItem`、`EchoChamber.onTriggered`。

## 玩家数据和能力背包

玩家 capability 集中在 `capability/player`：

- `PlayerPassiveItem`：额外被动道具背包，保存 `List<ItemStack>`；也维护套装计数 `Map<ResourceLocation, Integer>` 和已触发过的套装 `Set<Integer>`。
- `PlayerAbility`：攻击相关状态，包括 piercing/homing/spectral/controllable、额外饰品槽、蓄力值、attack type/bullet color/trajectory 计数和最佳结果缓存。
- `PlayerStatModifier`：保存各类属性 modifier 的实际值和显示值，并负责死亡后恢复属性。
- `PlayerItemPools`：玩家个人道具池移除/添加记录。
- `PlayerItemUseRecord`：最近使用的胶囊、卡牌、主动道具，以及胶囊 id 到效果 id 的识别记录。
- `PlayerSwallowedTrinkets`：吞下的饰品列表，并能合并 Curios 当前装备的饰品。

entity capability：

- `EffectModules`：挂在 `LivingEntity` 上，保存 `TriggerModuleQueue` 和 `RecursiveModuleQueue`。目前注释说主要用于玩家，避免性能问题。
- `EntityEffect`、`ExtraData` 也挂在 living entity 上，用于效果来源伤害、额外数据等。

能力数据的 attach/copy/sync 主要在 `event/ForgeEvents.java`：

- `AttachCapabilitiesEvent<Entity>` 给玩家和 living entity 添加 capability。
- `PlayerEvent.Clone` 死亡后复制各 capability。
- 登录时同步套装、胶囊识别、被动物品计数到客户端，并添加永久 trigger module。
- 每秒左右也会在 tick 中同步主要客户端数据。

## 胶囊系统

关键文件：

- `item/ModPills.java`
- `item/pickup/Pill.java`
- `item/pickup/GoldenPill.java`
- `manager/PillEffectManager.java`
- `manager/data/PillShuffleData.java`
- `registries/ability_effect/impl/pill_effect/PillEffect.java`

设计：

- `ModPills` 注册 13 个普通胶囊、13 个马胶囊、2 个黄金胶囊。
- 每个普通/马胶囊构造时向 `PillEffectManager` 注册 pill id。
- `PillEffectManager` 从 `executable_effect` registry 中筛出 `PillEffect` 类型，随机映射 `pillId -> effect ResourceLocation`。
- 映射保存在世界 SavedData `PillShuffleData`，世界加载时如果没有数据就 shuffle。
- 玩家使用未知胶囊后，服务端记录 pill id 对应的 effect id，并通过 `PillRecordsSyncS2CPacket` 同步到客户端。
- `PHD` / `False PHD` 会影响客户端 tooltip/name 展示，`PillEffect.redirect` 可根据玩家状态重定向效果。

注意：

- `ClientDataManager.isPillRecordCorrectly` 会拿客户端记录和 `PillEffectManager` 当前映射比较；多人/客户端启动时要保证世界映射已经正确同步或加载。
- `PlayerItemUseRecord` 中 `EffectRecords` 的 save/load 逻辑看起来写反了：`saveNBTData` 里像是在从 nbt 读，`loadNBTData` 里像是在往 nbt 写。后续若处理最近胶囊效果记录，应重点确认。

## 道具池和 loot

关键文件：

- `helper/PoolHelper.java`
- `capability/player/PlayerItemPools.java`
- `manager/data/ServerItemPoolsData.java`
- `loot/modifier/ItemPoolLootModifier.java`
- `loot/TempPoolManager.java`
- `manager/ModLootTables.java`
- 资源目录：`src/main/resources/data/isaac_disaster/loot_tables/pools/item`

设计：

- 道具池本质是 loot table。只有 namespace 是 `isaac_disaster` 且 path 以 `pools/item/` 开头的池会被动态处理。
- `ItemPoolLootModifier` 对生成出的 IsaacItem 做二次处理：按玩家/世界记录过滤已移除的道具，再加入玩家/世界额外添加的道具。
- 配置项决定是否从单池移除、全池移除、所有玩家共享池记录。
- 底座和箱子生成的道具会标记 removal；饰品通常不从池移除。
- README 提醒“从一个道具池一次生成多个物品可能不可预测”，代码里 `ItemPoolLootModifier` 也假设结果列表只有一个元素和原表只有一个 pool。

注意：

- `PlayerItemPools.getRemoval/getAddition` 和 `ServerItemPoolsData.getRemoval/getAddition` 返回的可能是内部 set，然后直接 `addAll(removeAll/addAll)`。这会把全局移除项写入单池 set，可能造成副作用。后续若重构道具池，这里优先检查。
- `TempPoolManager` 用玩家 UUID 暂存最近构造的 LootPool，并在 get 时 remove。多次嵌套/同 tick 多次生成时要小心覆盖。

## 底座、箱子、房间标记

底座：

- `block/custom/PedestalBlock.java`

## 触发型模块和循环型模块

触发型：

- 基类：`registries/trigger_module/TriggerModule.java`
- 队列：`TriggerModuleQueue` / `TriggerModuleInstance`
- 触发类型：`ModTriggerTypes`
- 注册：`ModTriggerModule`
- 事件派发：`event/TriggerModuleEvents.java`

设计：

- `TriggerModule` 内部持有 `CompositeTrigger`。
- `TriggerModuleEvents` 把 Forge/自定义事件转成 `ExecutableEffectContext`，设置 `EVENT`、`TARGET_POSITION`、`BULLET`、`SECONDARY_LIVING_ENTITIES`、`DOUBLE` 等上下文，再 dispatch。
- dispatch 时从 entity 的 `EffectModules` 里复制 trigger queue，按优先级执行。
- `GET_ATTACK_CONTEXT` 类型特殊：模块可通过 `attachToBullet` 给弹幕的 `AttackContext` 附加 trigger、颜色等。
- 玩家登录/死亡复制后会添加两个永久模块：`HIGH_PRIORITY_PLAYER_PERMANENT_MODULE` 和 `PLAYER_PERMANENT_MODULE`，用于常驻效果响应。

循环型：

- 基类：`registries/recursive_module/RecursiveModule.java`
- 队列：`RecursiveModuleQueue` / `RecursiveModuleInstance`
- 注册：`ModRecursiveModule`
- tick：`ServerTickEvent.recursiveModuleTick`

设计：

- 每 tick 对玩家的 recursive queue 调 `tickAll`。
- 每个实例维护 `stacks` 和 `coolDown`。
- 触发时构造 context，设置 `AMPLIFIER` 为 stacks、`TARGET_POSITION`、`RECURSIVE_MODULE_QUEUE`，然后 fire。
- interval 由模块按 entity/stacks/queue 动态返回。

注意：

- `TriggerModuleQueue.copy()` 和 `RecursiveModuleQueue.getCopy()` 当前都是浅拷贝实例列表。遍历期间如果实例字段被改，可能影响原队列或造成行为不直观。
- `TriggerModuleQueue.lock()` 只锁复制出来的 queue，并不锁实体上的真实 queue。设计上是防止本次 dispatch 的局部队列被改，不是全局并发锁。

关键文件：

- `registries/attack_type/AttackType.java`
- `registries/attack_type/ModAttackType.java`

- 子弹能力如 piercing/homing/spectral/controllable、颜色、轨迹都存在 `PlayerAbility` 或 `AttackContext` 中。

## 效果系统

关键文件：

- `registries/ability_effect/IExecutableEffect.java`
- `registries/ability_effect/IAbilityEffect.java`
- `registries/ability_effect/ExecutableEffectContext.java`
- `registries/ability_effect/ContextKeys.java`
- `CompositeTrigger` / `SimpleTrigger`
- `AbilityEffectEntry` / `CompositeExecutableEffect`
- `ModExecutableEffects.java`

设计：

- `ExecutableEffectContext` 是一个以 `ContextKey<T>` 为 key 的动态上下文 map，核心实体是 `LivingEntity entity`。
- `IAbilityEffect.apply` 默认跳过客户端，然后调用 `applyEffect`。
- `AbilityEffectEntry` 是“先写入参数，再调用另一个 effect”的包装。
- `CompositeExecutableEffect` 顺序执行多个 effect。
- `CompositeTrigger` 是一组 `SimpleTrigger`，可按 `TriggerType` fire。

代码习惯：

- 很多效果通过 `ContextKeys.BOOLEAN`、`DOUBLE`、`RESOURCE_LOCATIONS` 传 List 参数，约定依赖索引。
- 通用 effect 复用度高，但调用者必须知道每个 effect 读取哪些 key。
- `ModExecutableEffects` 很大，是“注册目录 + 部分配置包装”的混合体。以后找某个卡牌/主动/拾取的真实效果，先从这里追。

## 代码风格和习惯

- 很多系统使用 Forge `DeferredRegister` + `RegistryObject`。
- 大量实现类命名直观，通常一个以撒道具一个类。
- 逻辑经常分三层：Item 入口负责使用/装备/消耗，Ability 负责生命周期或 trigger，Effect/Module 负责可复用行为。
- 描述文本基本用 `Component.translatable`，实际文案在 lang json。
- 属性修改统一走 `StatManager`，避免每个道具直接写 AttributeModifier。
- 对“第一次获得/第一次装备/已消耗”的判断通过 ItemStack NBT 完成。
- 常用 helper 很集中：`PlayerHelper`、`LootHelper`、`LevelHelper`、`EntityHelper`、`CuriosHelper`、`PoolHelper`。
- 目前有不少注释和部分旧模板代码混在一起；实际改动时优先读调用链，不要只凭类名猜。

## 附属模组扩展性观察

比较好的扩展点：

- 自定义 registry 已经覆盖了 ability、effect、module、attack type、trajectory、bullet color 等核心概念。
- `ResourceLocation` 广泛用于 registry id、loot table id、set id，方向是对的。
- `IExecutableEffect` + `ExecutableEffectContext` 让外部效果可被 trigger 复用。
- loot table 和 tag 的使用让数据包层有一定扩展空间。

目前会限制扩展的地方：

- 道具/饰品的数字 id 来自 `ItemId` / `TrinketId` enum ordinal。外部模组无法自然加入这些 enum。
- 很多协同、tooltip、helper 查询依赖固定 int id，而不是 registry id 或 tag。
- `ItemId.ID_TO_ITEM` 需要手工 `registerItem`，并且当前看起来不是每个 ModPassiveItems 项都注册了 id->item 映射。道具池 addition 依赖它时可能不完整。
- `ModPassiveAbility`、`ModActiveAbility`、`ModExecutableEffects` 等是大静态注册表，外部能注册自己的对象，但项目内部很多逻辑仍只枚举本项目静态项。
- `ContextKeys` 使用通用 List 类型传参，对外部调用者不够自解释。

未来如果要强化附属模组支持，优先考虑：

- 用 `ResourceLocation` 或 Forge registry key 替代公共 API 里的 int id。
- 给道具能力暴露稳定接口，例如 `getAbilityId()`、`getIsaacItemId()`、`getQuality()`，减少 enum ordinal 泄漏。
- 把 “item id -> item registry object” 的注册做成自动扫描或统一 builder。
- 给 `IExecutableEffect` 增加小型文档/metadata，说明需要哪些 `ContextKeys`。

## 我看到的改进线索

这些不是本轮要修改的事项，只是后续遇到相关问题时优先检查：

- `PlayerItemUseRecord` 的 `EffectRecords` 保存/读取疑似反了。
- `PlayerItemPools` 和 `ServerItemPoolsData` 的 `getRemoval/getAddition` 可能修改内部 set。
- `GoldenPill.use` 注释是“5% 概率不消耗”，但代码是随机值 `< 0.05` 时 `shrink(1)`，行为像“5% 概率消耗”。
- `PlayerHelper.hasTrinket(..., onlyEnchanted)` 使用 `ItemStack::isEnchanted`，而项目饰品附魔状态似乎是 `Trinket.isEnchanted(stack)` NBT；这两者不一定一致。
- `PlayerHelper.getValueFromTrinket(normal, enchanted, ...)` 返回逻辑看起来可能把普通/附魔值反了：有附魔时返回 normal，没有附魔时返回 enchanted。
- `TriggerModuleQueue.copy()`、`RecursiveModuleQueue.getCopy()` 是浅拷贝；如果想要真正快照，需要复制 instance。
- `ItemPoolLootModifier` 假设 loot table 第一个 pool、结果只有一个物品；新增复杂 loot table 时要谨慎。
- `ModExecutableEffects` 过大，后续若频繁冲突，可以考虑按 active/pickup/pill/general 拆注册目录，但不要在小改动里贸然重构。
- `Config.java` 还保留 MDK 模板示例项，可在整理配置时清掉。
- 部分中文注释在终端默认编码下显示乱码，不代表源码坏了。

## 后续协作建议

- 如果任务是“新增一个普通被动道具”，先找一个同类型 implementation 复制模式，再补 `ItemId`、`ModPassiveAbility`、`ModPassiveItems`、lang、模型/贴图/loot pool。
- 如果任务是“新增主动道具”，重点在 `ActiveAbility` + `ModExecutableEffects` 组合；确认充能数在 `ModActiveItems`。
- 如果任务是“新增饰品”，确认效果是装备生命周期、事件触发、还是模块触发；很多饰品通过 `PlayerSwallowedTrinkets` 查询是否装备/吞下。
- 如果任务是“新增胶囊效果”，让类继承 `PillEffect` 并注册到 `ModExecutableEffects`；它会被 `PillEffectManager` 自动纳入 shuffle。
- 如果任务是“改道具池”，先读 `ItemPoolLootModifier` 和 `PoolHelper`，再改 loot table，不要只改资源文件。
- 如果任务是“改攻击/弹幕”，先读 `AttackType`、`PlayerAbility`、`TriggerModuleEvents.getAttackContext`，因为很多道具是在 attack context 阶段注入行为。
- 如果任务涉及客户端 tooltip/HUD，别忘了同步包和 `ClientDataManager`。
